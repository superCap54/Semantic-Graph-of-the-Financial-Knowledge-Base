# 产品规格：金融知识库语义图谱

## 1. 产品定位

本项目是一个通用金融知识库语义图谱，用于持续研究和跟踪一个行业或某只股票。

它不是普通新闻收藏夹，而是一个由 AI 维护的 Obsidian 知识库：

- 收集多源信息。
- 保存原始证据。
- 分析每条信息的可信度和影响。
- 抽取公司、行业、事件、政策、产品之间的关系。
- 生成日报、周报和长期分析日志。

第一验收样板为：`华大基因（300676.SZ）`。

## 2. 使用目标

用户输入或配置一个关注对象，例如 `华大基因（300676.SZ）`，系统应持续收集并归档：

- 公司新闻。
- 行业新闻。
- 上中下游产业链信息。
- 相关政策。
- 竞争对手新闻。
- 财报、公告、监管文件。
- 产品和业务线信息。
- 相关事件。
- 社媒、论坛、自媒体和市场讨论。

系统同时输出：

- 事实层：来源可追溯的 raw 资料和结构化事件。
- 分析层：AI 对每条信息和周期信息的判断与结论。

## 3. Vault 与 Git 边界

目录 `Semantic-Graph-of-the-Financial-Knowledge-Base/` 本身同时是：

- 独立 Obsidian vault。
- GitHub 仓库。
- Codex/AI 读写和维护的金融知识库。

所有知识内容、代码、配置和 `.obsidian/` 配置默认纳入 Git 版本管理。

## 4. 核心目录

```text
raw/
  horizon-reports/
  news/
  announcements/
  policies/
  reports/
公司/
行业/
事件/
政策/
产品/
日报/
周报/
索引/
```

### raw

证据层。

- `horizon-reports/`：保留 Horizon 原始 Markdown 报告。
- `news/`：从 Horizon 报告拆出的单条新闻卡片。
- `announcements/`：公告、财报、交易所问询函、公司公开文件，尽量完整保存。
- `policies/`：政策原文、政府文件、监管文件，尽量完整保存。
- `reports/`：研报，默认保存摘要和关键摘录；不默认保存全文。

### 公司 / 行业 / 事件 / 政策 / 产品

Obsidian 五类核心节点。

这些文件夹不是普通收纳盒，而是五种节点类型。系统要围绕它们维护关系：

- 公司属于哪个行业。
- 公司生产什么产品。
- 公司或产品卖给谁、进入什么场景。
- 事件影响了谁。
- 政策作用到哪个行业、产品和公司。

### 日报 / 周报

- 日报：信息流汇总 + 今日重要变化。
- 周报：本周变化 + 累计认知是否发生变化。

## 5. 文件命名

- 公司：`公司名（股票代码）.md`，例如 `华大基因（300676.SZ）.md`。
- 行业：自然中文名，例如 `基因检测行业.md`。
- 政策：自然中文名，例如 `医疗器械监管政策.md`。
- 产品：自然中文名，例如 `肿瘤基因检测.md`。
- 事件：日期 + AI 自动事件名，例如 `2026-07-16 华大基因半年报及市场解读.md`。

## 6. 证据保存策略

采用混合策略：

- 官方公告、财报、交易所文件、监管政策、政府文件、公司官网资料：尽量完整保存。
- 商业媒体新闻、普通新闻网页、自媒体、论坛、社媒：保存链接、摘要、关键摘录和结构化记录。
- 研报、付费内容、版权不明确材料：默认不保存全文。

原则：

- raw 永久保留，作为证据来源。
- 所有分析必须能回链到 raw。
- 若证据不足，必须标记“证据不足”或“待验证”。

## 7. 采集策略

第一步接 Horizon。

Horizon 作为独立信息采集器运行，本项目读取 Horizon 输出的 Markdown 报告。

后续采集源通过 adapter 扩展：

- Horizon。
- 东方财富。
- 雪球。
- 巨潮资讯。
- 交易所。
- 公司官网。
- 政策网站。
- RSS。
- 搜索引擎。

所有 adapter 最终输出统一 raw 卡片格式，后续分析层不关心来源。

## 8. Horizon MVP 流程

```text
Horizon Markdown 报告
-> raw/horizon-reports/
-> AI 拆成多条 raw/news/ 新闻卡片
-> AI 对每条新闻评分和分析
-> 宽松合并重复新闻为主事件
-> 更新 事件/公司/行业/政策/产品
-> 生成 日报/
```

## 9. 单条信息分析模板

每条信息至少包含：

```md
## 单条分析

### 事实摘要

### 涉及对象

### 关系抽取

### 可信度判断

### 影响分析

### 短期/中期/长期影响

### AI 结论
```

## 10. AI 评分

每条信息用 10 分制，不使用“低/中/高”作为主评分。

```md
## AI 评分

- 重要性评分：8/10
  理由：...

- 可信度评分：6/10
  理由：...

- 影响强度评分：7/10
  理由：...
```

评分含义：

- 重要性评分：是否值得进入长期知识库。
- 可信度评分：能否作为事实依据。
- 影响强度评分：对公司、行业、产品、政策判断的影响程度。

## 11. 可信度分层

所有信息都可以进入分析，但必须标注可信度。

```yaml
source_tier: official | mainstream | social | self_media | forum | unknown
evidence_type: announcement | filing | policy | news | opinion | rumor
credibility_score: 1-10
```

处理原则：

- 官方公告和政策可以作为强事实。
- 主流媒体作为较强线索。
- 社媒、论坛、自媒体可以进入分析，但必须明确标注为观点、传闻或未验证线索。
- 低可信信息不能直接写成事实结论。

## 12. 去重与事件合并

事件去重采用宽松合并。

同一主题下的多篇报道只生成一个主事件，其他报道挂到该事件的来源或补充来源部分。

例如：

```text
华大基因发布半年报
媒体解读华大基因半年报
雪球讨论华大基因利润变化
自媒体分析华大基因检测业务承压
```

可合并为：

```text
事件/2026-08-28 华大基因半年报及市场解读.md
```

## 13. 长期文件更新

长期文件包括公司、行业、政策、产品。

更新策略：

- 重要事件单独追加。
- 普通事件每日合并追加。
- 不采用覆盖式维护，保留认知变化历史。
- 文件超过 2MB 或季度检查时再归档旧日志。

## 14. 日报

日报用途：

- 信息流汇总。
- 今日重要变化简报。

日报不负责引导用户思考，不做操作台。

建议结构：

```md
# 2026-07-16 日报

## 今日重要变化

## 今日新增资料

## 今日可信度分布

## 今日分析结论
```

## 15. 周报

周报用途：

- 总结过去 7 天新增变化。
- 结合历史资料判断累计认知是否变化。

建议结构：

```md
# 2026-W29 周报

## 本周重要变化

## 本周新增资料统计

## 公司层面更新

## 行业层面更新

## 政策层面更新

## 产品/业务层面更新

## 可信度与待验证信息

## 累计认知更新

## AI 综合结论
```

## 16. Watchlist

系统以长期关注清单为主。

配置文件内部字段使用英文，Obsidian 输出使用中文。

用户只需输入关注对象，AI 自动补全能确定的信息，需要人工确认的内容再由用户补充。

默认追踪范围偏广，包括：

- 公司本体。
- 别名和股票代码。
- 所属行业。
- 上中下游。
- 产品和业务。
- 竞品。
- 关键客户和供应商。
- 相关政策。
- 宏观变量。

## 17. 开发节奏

第一阶段：分步骤 CLI，方便调试。

```text
collect -> normalize -> analyze -> compile -> daily
```

第二阶段：稳定步骤串联。

```text
sgkb run <关注对象>
```

第三阶段：定时任务。

```text
sgkb run-watchlist --daily
```
