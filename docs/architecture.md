# Architecture

## 目标

把金融信息从“新闻列表”变成“可追溯、可增长、可追问的 Obsidian 语义图谱”。

## 分层

### 1. Ingest layer

负责把外部资料拉回本地：

- 财经新闻 API
- RSS
- 公告、交易所、监管网站
- 研报、PDF、网页剪藏
- 类 Horizon 的聚合源
- 手动粘贴材料

输出只进入 `knowledge/raw/`，不直接污染 wiki。

### 2. Normalize layer

把不同来源统一成一个本地 Markdown 记录：

```yaml
---
title: 原始标题
source: 信息源
url: 原文链接
captured_at: 2026-07-16T00:00:00+08:00
query: 贵州茅台
status: raw
---
```

正文保留原始摘要、正文、图片路径、抓取备注。

### 3. Compile layer

由 LLM/Codex 执行增量编译：

- 抽取实体：公司、行业、产品、人物、机构。
- 抽取事件：财报、并购、政策、价格变化、产品发布、风险事件。
- 建立关系：`公司 -> 所属行业`、`产品 -> 公司`、`政策 -> 影响行业`、`事件 -> 涉及公司`。
- 更新 wiki 词条。
- 维护索引页和待研究问题。

### 4. Obsidian layer

Obsidian 只负责浏览、回链、图谱、人工快速检查。主要写入者是 LLM。

### 5. Tool layer

本地 CLI 和未来 Web UI/API 负责：

- 一键抓取某个主题。
- 查看待编译资料。
- 搜索 wiki。
- 触发 LLM 编译。
- 做健康检查：断链、重复词条、缺来源、矛盾结论。

## 为什么第一版不用数据库

文件系统足够支持早期闭环：

- Git 能版本化每次知识库增长。
- Obsidian 能直接消费 Markdown。
- Codex 能直接读写目录。
- 小规模知识库可以靠索引页和全文搜索工作。

未来需要数据库的信号：

- 原始资料超过数万篇。
- 需要复杂图查询。
- 需要多人同时写入。
- 需要向量检索、权限、任务队列或可视化性能。

## Horizon 的可借鉴点

Horizon 这类系统的价值不在“某一个 API”，而在这几个流水线能力：

- 多源抓取。
- 并发执行。
- 去重。
- 相关性评分。
- 摘要和背景补全。
- 定时生成 digest。

本项目可以借用这个思路，但输出目标改成 Obsidian wiki，而不是只生成新闻简报。
